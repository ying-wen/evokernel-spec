# v1.2 Vision: Any-Model × Any-Hardware Compilation & Optimization

> **Status:** design — landing in v1.2 (target Q3 2026). Drives concrete
> work tracked in [ROADMAP.md](ROADMAP.md).

## Why we're broadening scope

v1.0 / v1.1 framed the project narrowly: "LLM inference knowledge
base, with comprehensive Chinese accelerator coverage". That was right
for bootstrapping — LLMs have a clean shape (decoder-only Transformer,
roofline-friendly, well-studied) and are the largest current spend.

But the deeper value is the **kernel** under that scope: a
schema-validated, evidence-backed corpus mapping models → hardware →
deployment outcomes, with a transparent calculator for portability.
That same kernel can carry far more than chat LLMs. Specifically,
the project should evolve into:

> **A computable knowledge platform for compiling and optimizing
> ANY model onto ANY accelerator** — including scientific / HPC /
> simulation models, vision/speech, and niche/edge hardware that
> mainstream "GPU benchmark" sites ignore.

This document captures the design of that broadening, the schema
implications, the hardware/model corpus expansion, and the
visualization upgrades.

## What changes

### 1. Model corpus — beyond LLMs

Today every model in `data/models/` is a transformer-family LLM. v1.2
adds a `domain` field to `ModelSchema` and seeds:

| Domain | Examples (priority) |
|---|---|
| `llm` (existing) | DeepSeek V4, Llama 4, Qwen 3.5, GLM 5, ... |
| `scientific` | AlphaFold 3 (protein folding), GraphCast (weather), AlphaProteo, ESMFold, NeuralGCM |
| `multimodal` | Llama 4 Scout-V, Gemma 4 vision, Qwen-VL 3 |
| `speech` | Whisper v3, Distil-Whisper, ParlerTTS, Sense Voice |
| `vision` | SAM 2, DINOv3, Grounding DINO, RT-DETR |
| `recommender` | DLRM, Wide&Deep variants — large embedding tables, distinct compute profile |
| `code` | DeepSeek Coder V3, Qwen Coder 3 |

Each domain has different roofline characteristics:

- **Scientific** (AlphaFold/GraphCast): irregular memory access,
  graph neural network ops, often FP32 or mixed precision, latency
  sensitive (single-shot inference for a folded protein).
- **Speech**: streaming, very low TBT requirement (real-time captions),
  encoder-decoder split, attention chunking.
- **Vision**: 2D/3D conv-heavy, batch sizes from 1 (RT) to 1024 (offline
  re-tagging).
- **Recommender**: embedding-table-bound (TBs of memory bandwidth needed
  for sparse lookups), small dense net.
- **Code**: same as LLM but FIM-mode prefill is structurally different.

The calculator extends to handle these, with operator decomposition
expanded beyond the current attention/MLP/MoE set.

### 2. Hardware corpus — niche + frontier

Today: 31 cards across 9 China + 5 Western vendors, all general-purpose
data-center accelerators. v1.2 adds:

**Frontier (mentioned by user):**
- NVIDIA Rubin / Vera Rubin (announced Computex 2025; estimated specs)
- Huawei Ascend 950 series (announced 2025 roadmap)
- AMD MI400X (CDNA 5 — announced)
- Intel Gaudi 4 / Falcon Shores

**Niche / specialized inference:**
- **Cerebras WSE-3** — wafer-scale, 4 trillion transistors, 900,000 cores,
  44 GB on-die SRAM. Different roofline math entirely (no HBM, single die).
- **Groq LPU** — deterministic latency via static scheduling, 230 MB SRAM
  per chip, no DRAM. Ideal for batched-1 LLM serving.
- **SambaNova SN40L** — RDU (Reconfigurable Dataflow Unit) with HBM3 +
  DDR5 + 64MB on-chip. Spatial dataflow.
- **Tenstorrent Wormhole / Blackhole** — open-source RISC-V-based,
  Galaxy interconnect.
- **Etched Sohu** — transformer-only ASIC; 100k+ tok/s claimed for
  Llama 70B.
- **Modular MAX / Mojo** — software-defined accelerator targeting
  CPUs+GPUs uniformly.

**Edge inference:**
- **Hailo-10** — 40 TOPS, 5W, automotive ADAS
- **Qualcomm Cloud AI 100 Ultra** — 870 TOPS @ 75W, edge inference
- **Apple M4 Max NPU** — 38 TOPS, on-device inference
- **Rockchip RK3588 NPU** — 6 TOPS, embedded SBC

**Scientific computing accelerators (HPC):**
- **NVIDIA Grace Hopper GH200** (already partially covered — extend
  with HPC-specific FP64 metrics)
- **Cerebras CS-3** (cluster form of WSE-3, oriented at scientific)
- **AMD MI300A** (APU variant — scientific simulation focus)
- **NEC SX-Aurora TSUBASA** — vector processor, fluid dynamics

The schema accommodates these natively:

- `form_factor` extends: `wafer-scale | edge | embedded | apu | vector`
- `compute` adds optional `fp64_tflops` (HPC), `tops_int4` (edge)
- `memory.type` extends: `on-die-sram | on-package-sram | hbm-on-package`

### 3. Topology — show what the chip actually does

The current Topology component renders a die illustration with generic
SM/CU blocks. v1.2 makes it **actively informative** about the
intersection of (hardware × model × operator):

**New overlays:**

1. **Operator-fitness heatmap** — given a selected model, color each
   compute unit by which operator dominates its time (attention vs
   MLP vs conv vs scatter/gather). Reveals e.g. "this card is
   attention-bound for DeepSeek V4 but compute-bound for AlphaFold".

2. **Memory pressure overlay** — show KV cache footprint (LLM),
   activation memory (scientific), embedding table residency
   (recommender) as colored bands on the HBM stack glyphs.

3. **Cluster zoom** — hover any card to see "if you scale this to 8 /
   72 / 384 cards, here's the scale-up domain layout, scale-out NIC
   placement, and bisection bandwidth". Surfaces NUMA / topology
   constraints for collective ops.

4. **Compilation path annotation** — for each operator in the
   selected model, show the compilation chain it goes through:
   `aten::linear → cublas | torch.compile → triton | TVM → cuda`.
   Click to see the kernel signature.

5. **Critical path highlight** — for a given model + hardware, the
   pipeline of ops on the critical path is highlighted across the
   topology, with per-op latency labels.

These overlays compose: select model → select operator → cluster
zoom shows cross-card operator scheduling.

### 4. Calculator → Compilation Optimizer

The calculator becomes a **portability advisor** that suggests:

- "DeepSeek V4 on Ascend 910C is 0.18 calibrated efficiency. Top 3
  optimization wedges with biggest predicted lift:
   1. Switch GEMV to FlashAttention-v3 (predicted +35%, evidence:
      similar lift on H100 case-001)
   2. Tile size 128→64 for MoE up-projection (memory-bound; +12%)
   3. Disaggregate prefill/decode (latency floor 80%; +8% throughput)"
- Each suggestion is rooted in a real case in the corpus, not a
  hand-written rule.

This is the **knowledge graph payoff**: 22 deployment cases form a
portability oracle, surfaced through the calculator UI.

### 5. Schema additions

```typescript
// schemas/model.ts
domain: z.enum([
  'llm', 'multimodal', 'vision', 'speech', 'scientific',
  'recommender', 'code', 'rl', 'graph'
]).default('llm');

workload_kind: z.enum([
  'autoregressive-decode',
  'encoder-only',
  'encoder-decoder',
  'streaming',
  'graph-iteration',     // GraphCast, MD simulation
  'forward-only-batch',  // single-shot scientific inference
  'embedding-table-lookup' // recommender
]);

// schemas/hardware.ts
compute: {
  // ... existing tflops fields
  fp64_tflops: ValueWithEvidence().optional(),  // HPC scientific
  fp32_tflops: ValueWithEvidence().optional(),  // mixed precision
  tops_int4: ValueWithEvidence().optional(),    // edge inference
}

architecture: {
  // ... existing fields
  on_die_sram_mb: ValueWithEvidence().optional(),  // Cerebras / Groq
  reconfigurable: z.boolean().optional(),          // SambaNova RDU
  deterministic_latency: z.boolean().optional(),   // Groq LPU
}

form_factor: z.enum([
  // ... existing pcie/sxm/oam
  'wafer-scale',
  'edge-m2',
  'embedded-soc',
  'apu',
  'vector-card'
]);
```

### 6. New pages

- `/scientific` — counterpart to `/china`: matrix of HPC models ×
  hardware fitness, with calibration data from MLPerf HPC + AlphaFold
  benchmark suite
- `/edge` — embedded accelerators with TOPS/W ranking, automotive
  certifications, latency-bounded benchmarks
- `/compile` — compilation chain explorer: pick a model + a target
  hardware, see the path through PyTorch → IR → kernel library
- `/operators/[op]` (existing, expanded) — per-operator hardware
  fitness with explicit decomposition into compute / memory / synch
  components

## Non-goals (stay focused)

- **We do NOT become an inference engine** — no kernels, no runtime.
  We point at vLLM, TensorRT-LLM, MindIE, etc.
- **We do NOT host training** — strictly inference / forward-pass.
- **We do NOT become a hardware seller** — vendor-neutral, evidence-based.
- **No proprietary benchmark scripts** — every claim cites a public
  reproducible source.
