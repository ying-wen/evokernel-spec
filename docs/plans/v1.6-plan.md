# v1.6 Iteration Plan — proposed

**Status:** draft awaiting user confirmation
**Drafted:** 2026-04-30 (post v1.5.1)
**Estimated effort:** 1.5–2.5 days of focused work

---

## TL;DR

v1.5 closed the convergence loop: every model page now has a 3-leaderboard "推荐硬件" widget. v1.6 makes the same widget work the **other direction** (hardware → models), thickens the long-tail data so recommendations have solid foundations across all 39 cards (not just 7), and fills a few obvious holes in the fused-kernel catalog. One-shot release with 4 cohesive deliverables.

---

## Goal

> 让每个 /hardware/<slug>/ 页面也变得"对决策有用" — 用户带着一张已有的卡来网站，能立刻看到"在这张卡上跑哪些模型最划算/最快/有实测数据"。

This mirrors v1.5 (which answered "given a model, what hardware?") with the symmetric question (given hardware, what model?). The recommendation engine and TCO formula are already done; v1.6 is mostly UX wiring + data backfill.

## Why this is the right next iteration

1. **Symmetric move beats novel feature.** v1.5's recommendation engine has 6 unit tests + production usage on 19 model pages. Reusing it for the reverse direction is low-risk, high-leverage.
2. **Data density catches up to schema density.** v1.3–v1.4 built rich `memory_hierarchy` / `cluster-internals` / `fused-kernels` schemas. Currently filled for 7 / 2 / 8 entities respectively. v1.6 brings these to 12 / 5 / 12 — the recommendation widget gets richer foundation.
3. **No new schema work.** Everything in v1.6 uses existing fields. Pure value extraction from existing infrastructure.
4. **User-flagged gap #3 (hardware/cluster internal) keeps deepening.** This is the gap with the longest tail.

**Effort:** ~1 day

The `recommendModelsForHardware()` helper + `ReverseRecommendationRow` type already shipped in v1.5.0 commit `6cdcbb1`. Just need to wire UI:

- [ ] `apps/web/src/components/hardware/RecommendedModels.astro` — 3-leaderboard widget

  - 💰 lowest $/M tokens (uses measured throughput when case exists for this hw × model)
  - ✅ verified by measured cases (filtered to hw+model combos with real data)
  - Each row: rank #, model name, lab, params (active/total), precision used for scoring, bottleneck tag, calibrated badge, throughput/cost figure, "tune →" deep-link to `/calculator/?model=...&hw=...`

2. **B + C** in parallel — pure YAML authoring, no code changes, low coordination overhead
3. **D** last — the fused kernels are independent of A/B/C and a natural last-step polish
4. Tag v1.6.0; deploy; verify production
