# HW-Software Layer Gap — 5-Layer Decomposition

> **Status**: design supplement to `2026-05-02-agent-readiness.md` · 2026-05-02
> **Purpose**: decompose the cross-vendor hw-software gap into 5 specific layers
> with concrete schema additions, and show how they compose for a deployment agent.

---

## Why this supplement exists

The earlier agent-readiness design doc identified "Cross-hardware op equivalence" as **the BIGGEST GAP** but treated it abstractly. The user pushed back specifically:

> 还得思考从模型本身架构/参数到任意硬件除了硬件本参数/集群参数外，硬件软件层面的GAP，包括CUDA算子和实现，CANN和类似其他算算子实现语法不同，功能不同，覆盖度不同等等问题。

Translated: *"Beyond hardware specs, the hw-software-layer gap matters: CUDA operators and implementations vs CANN-and-similar — different syntax, different functionality, different coverage."*

This document answers **how to model that gap structurally** so the agent can navigate it.

---

## The 5 layers of hw-software heterogeneity

A kernel that runs on H100 vs Ascend 910C vs MI300X differs at **five distinct layers**, each requiring its own representation in the corpus:

```
┌───────────────────────────────────────────────────────────────┐
│ Layer E: Coverage Matrix                                      │ — which ops × which libs × which precisions
│ ─────────────────────                                         │   are actually shipped
│                                                               │
│ Layer D: Functional Semantics                                 │ — does softmax handle -inf the same way?
│ ─────────────────────────                                     │   is the reduction order deterministic?
│                                                               │
│ Layer C: Operator Library (cuBLAS / aclnn / rocBLAS / ...)    │ — opaque high-level APIs
│ ──────────────────────────                                    │
│                                                               │
│ Layer B: Programming Model (CUDA / Ascend-C / HIP / MUSA / Metal) │ — kernel languages, memory models
│ ───────────────────────────                                       │
│                                                               │
│ Layer A: ISA Primitives (WGMMA / MFMA / Cube / WMMA / AMX)    │ — actual tensor instructions
│ ────────────────────────                                      │
└───────────────────────────────────────────────────────────────┘
```

Today the corpus has Layer C *partially* (`engine_implementations[].hardware_arch` tags) and **nothing** on layers A, B, D, E.

## Layer A — ISA Primitives

What it captures: the actual matrix / vector / tensor instructions the silicon exposes.

| Vendor / Arch | Primary tensor instruction | Operand sizes | Async? |
|---|---|---|---|
| NVIDIA Hopper | WGMMA `m64nNk16` (warp-group MMA) | M=64, N=8..256 step-8, K=16/32 | ✅ |
| NVIDIA Blackwell | WGMMA + TCGEN05 + FP4 path | M=64..256, K=16..64 | ✅ |
| NVIDIA Ampere | WMMA / mma.sync | 16x16x16, 16x8x8 | ❌ |
| AMD CDNA3 | MFMA `32x32x16`, `16x16x32` | shape-quantized | ✅ |
| AMD CDNA4 | MFMA `16x16x32` (FP4 path) | shape-quantized + FP4 | ✅ |
| Huawei Ascend 910C | Cube unit (16x16x16 base) + Vector + Scalar | 16x16x16 fp16/bf16 | ✅ |
| Cambricon MLU | Matrix-multiply unit (proprietary) | 16x16x16 typical | ✅ |
| Apple M-series | AMX (Apple Matrix Extension) | 32x32 fp16 | ✅ |

**Schema proposal** — `data/isa-primitives/<id>.yaml`:

```yaml

vendor: nvidia
arch_family: hopper
class: tensor-mma     # categories: tensor-mma | matrix-vector | reduction | scan | conv | special-fn
name: WGMMA (Warp Group Matrix Multiply Accumulate)

shapes_supported:
  - { M: 64, N: 64, K: 16, dtype_a: fp16, dtype_b: fp16, dtype_c: fp16 }
  - { M: 64, N: 128, K: 16, dtype_a: fp16, dtype_b: fp16, dtype_c: fp16 }
  - { M: 64, N: 256, K: 16, dtype_a: fp8-e4m3, dtype_b: fp8-e4m3, dtype_c: fp32 }
  # ... full table

memory_model:
  operand_a_source: shared-memory  # vs registers / global
  operand_b_source: shared-memory
  result_destination: registers
  async: true                      # async vs blocking
  requires_tma: true               # TMA descriptor required for operand load

calling_convention:
  asm_intrinsic: "wgmma.mma_async.sync.aligned.m64nNk16"
  cuda_intrinsic_header: "<cuda/std/mma>"
  cutlass_template: "cutlass::arch::Sm90"

cross_vendor_equivalents:
  # The keystone field — primitive-to-primitive mapping for porting
  - vendor: amd
    arch_family: cdna3
    primitive_id: amd-cdna3-mfma-32x32x16
    notes: "MFMA 32x32x16 → 32M tile; 2 MFMAs to match WGMMA m64."
  - vendor: huawei
    arch_family: ascend-910c
    primitive_id: huawei-ascend-cube
    notes: "Cube does 16x16x16 → 4 cube ops to match WGMMA m64n64k16 fp16. Async via OE pipeline."
  - vendor: amd
    arch_family: cdna4
    primitive_id: amd-cdna4-mfma-16x16x32-fp4
    notes: "FP4 native; matches Blackwell's WGMMA FP4 path."

used_by_kernels:
  - flash-attention-v3   # operator-level reference
  - paged-attention
  # references back to fused-kernel ids
```

## Layer B — Programming Model / Kernel Language

What it captures: the language an engineer writes a custom kernel in.

| Language | Vendor | Memory model | Sync primitives | Compiler |
|---|---|---|---|---|
| CUDA C++ | NVIDIA | Block + grid + shared memory + register | `__syncthreads`, `__threadfence`, atomic | nvcc |
| HIP | AMD | Same as CUDA (1:1 source compat) | hipDeviceSynchronize, atomic | hipcc / amdclang |
| OpenAI Triton | Multi-vendor | Block-tile abstraction over SPMD | barrier, atomic | triton compiler → PTX/AMD |
| Ascend-C / CCE-C | Huawei | **GM / UB / L1 / L0 hierarchy** (explicit) | pipeline OE / OQ | bisheng / dav-c |
| BANG-C | Cambricon | NRAM / WRAM / SRAM (explicit) | __sync_cluster, __sync_local | cncc |
| MUSA-C | Moore Threads | CUDA-derivative (similar to HIP→CUDA) | musaDeviceSynchronize | mucc |
| Metal Shading Language | Apple | Threadgroup + tile | threadgroup_barrier | xcrun metal |

**Critical differences agents must know about**:
- **Memory hierarchy**: CUDA presents L1/SMEM/registers as a unified abstraction; **Ascend-C requires explicit DMA between GM↔UB↔L1↔L0**. A kernel that "just works" on CUDA may need 3-stage DMA pipelining on Ascend.
- **Async patterns**: CUDA uses streams + events; Ascend uses OE/OQ pipelines; HIP mirrors CUDA; Triton abstracts both behind `tl.async_copy`.
- **Sync primitives**: subtle semantics differences in memory ordering can break correctness when ported.

**Schema proposal** — extend `VendorSchema` with `software_stack`:

```yaml
# data/vendors/huawei.yaml extension
software_stack:
  kernel_languages:
    - id: ascend-c
      name: Ascend-C
      paradigm: spmd-with-explicit-memory-pipeline
      memory_hierarchy_levels: [GM, UB, L1, L0]
      async_pattern: pipeline-oe-oq
      compiler: bisheng
      runtime: ACL
      docs_url: "https://www.hiascend.com/document/detail/zh/ascend-c-api/"
      example_kernel_repo: "https://gitee.com/ascend/samples"
    - id: cce-c
      name: CCE-C (legacy)
      status: deprecated
      successor: ascend-c

  ai_frameworks:
    - { id: mindspore, native: true }
    - { id: pytorch, integration: torch_npu }
  ops_libraries: [aclnn, atb, opskernel-builtin]

  porting_caveats:  # specific things that bite when porting from CUDA
    - "Memory hierarchy is explicit (UB/L1/L0) — kernel must DMA in 3 stages"
    - "Pipeline parallelism via OE/OQ instead of streams"
    - "Cube/Vector/Scalar units are physically separate — cross-unit ops cost cycles"
```

**Concrete v2.7 deliverable**: `software_stack` populated on all vendors with a kernel programming language (NVIDIA / AMD / Huawei / Cambricon / Moore Threads / Apple / Intel / Tenstorrent).

## Layer C — Operator Library (cuBLAS / aclnn / rocBLAS / CNNL)

What it captures: the high-level opaque APIs that ship most production ops without writing a kernel.

| Library | Vendor | API style | Op coverage | FP8 support |
|---|---|---|---|---|
| cuBLAS / cuBLASLt | NVIDIA | opaque handle | gemm / gemv / batched-gemm | ✅ Hopper+ |
| cuDNN | NVIDIA | opaque + workspace | conv / attention / norm / activation | ✅ Hopper+ |
| CUTLASS | NVIDIA | C++ template | full GEMM family + attention | ✅ |
| rocBLAS | AMD | opaque (cuBLAS API-compat layer) | gemm-family | ⚠️ partial |
| MIOpen | AMD | opaque (cuDNN-like) | conv / norm / activation | ⚠️ |
| CK (Composable Kernel) | AMD | C++ template (CUTLASS-like) | gemm + attention + conv | ✅ MI300X+ |
| aclnn | Huawei | **workspace + executor pattern** | gemm / attention / norm / softmax / reduce | ⚠️ partial 910C+ |
| ATB (Ascend Transformer Boost) | Huawei | high-level transformer ops | full transformer block | ⚠️ |
| CNNL | Cambricon | opaque (similar to cuDNN) | wide neural-net coverage | ⚠️ partial |
| MPSGraph | Apple | graph-based | tensor ops via Metal | ❌ FP4 unsupported |

**Critical "covers different things"**:
- cuDNN + cuBLAS together cover ~80% of LLM inference ops natively on NVIDIA
- aclnn covers ~70% of those on Ascend, with ATB filling transformer-specific gaps
- rocBLAS+MIOpen cover ~75% on AMD; CK fills the rest for newer ops
- Cambricon CNNL ~65%; rest needs custom BANG-C kernels

**Schema proposal** — new entity `data/kernel-libraries/<id>.yaml`:

```yaml
id: aclnn
name: aclnn (Ascend C Neural Network library)
vendor: huawei
language: ascend-c     # references kernel-language id
api_style: workspace-executor   # vs opaque-handle / template

target_archs: [ascend-910b, ascend-910c, ascend-910d]

covers_op_classes:
  - { class: gemm, coverage: full }
  - { class: gemv, coverage: full }
  - { class: attention, coverage: partial, notes: "Standard MHA + GQA; MLA via separate ATB op" }
  - { class: layer-norm, coverage: full }
  - { class: rms-norm, coverage: full }
  - { class: softmax, coverage: full, notes: "Fused with mask via aclnnMaskedSoftmax" }
  - { class: rope, coverage: full }
  - { class: moe-gating, coverage: experimental, notes: "910D path mature; 910C requires custom kernel" }
  - { class: speculative-verify, coverage: missing }

precision_support: [fp16, bf16, int8, int4-awq]    # FP8 via separate atb op only

abi_signature_pattern: |
  aclnnXxxGetWorkspaceSize(args, &workspaceSize, &executor);
  aclnnXxx(workspace, workspaceSize, executor, stream);

include_paths: [aclnnop/aclnn_xxx.h]
linker_flags: [-lopapi, -lascendcl]

cross_vendor_equivalents:
  - { lib: cublas, op_class: gemm, equivalence: 'cublasGemmEx ↔ aclnnMatmul', notes: "Different API surface; 1:1 functionally" }
  - { lib: cudnn, op_class: layer-norm, equivalence: 'cudnnLayerNormForward ↔ aclnnLayerNorm' }

docs_url: "https://www.hiascend.com/document/detail/zh/canncommercial/..."
maturity: stable
```

Plus extend `OperatorSchema.engine_implementations[]` with a `kernel_library: <slug>` field so the existing structure can point at these new entities.

**Concrete v2.5 deliverable**: 8 kernel-library entries populated (cuBLAS / cuDNN / CUTLASS / rocBLAS / MIOpen / CK / aclnn / CNNL) and operator implementations cross-referenced.

## Layer D — Functional Semantics

What it captures: even when an op exists in two libraries, the *semantics* may differ subtly.

**Examples that break agent codegen**:
- **softmax with `-inf` input**: cuDNN returns `0`; some Triton implementations return `NaN`. If agent ports softmax from CUDA to CANN without checking, attention masks may silently misbehave.
- **layer-norm reduction order**: cublas guarantees deterministic reduction; rocBLAS pre-MI300X did not (parallel reduction non-deterministic). Affects training reproducibility.
- **FP8 scaling rules**: NVIDIA per-tensor scaling vs MXFP4 per-32-block scaling vs NVFP4 per-16-block scaling — same dtype name, different semantics.
- **mixed-precision matmul accumulation**: cuBLAS-FP16 accumulates in FP32 by default; some libs accumulate in FP16 → silent quality drop.
- **Empty-tensor behavior**: PyTorch returns empty tensor; some custom kernels SEGFAULT; CUTLASS asserts.

**Schema proposal** — extend `OperatorSchema` with `formal_semantics`:

```yaml
# Extends data/operators/softmax.yaml
formal_semantics:
  signature: |
    softmax(x: Tensor[..., D], dim: int = -1) -> Tensor[..., D]
    output[..., i] = exp(x[..., i] - max(x)) / sum_j(exp(x[..., j] - max(x)))

  edge_cases:
    - input: "all -inf"
      cublas: "0 in all positions"
      cudnn: "0 in all positions"
      triton-default: "NaN (no max-subtract default)"
      aclnn: "0 in all positions"
      mitigation: "Always use stable variant (max-subtract); check Triton version"

    - input: "single element"
      all_libs: "1.0"

    - input: "empty dim"
      cublas: "empty tensor"
      pytorch: "empty tensor"
      cutlass: "ASSERT — refuses to dispatch"

  numerical_rules:
    deterministic_reduction:
      - { lib: cublas, deterministic: true }
      - { lib: cudnn, deterministic: true }
      - { lib: triton-flash-attn-3, deterministic: false, notes: "Parallel reduction order varies" }
      - { lib: rocblas, deterministic: { default: false, override: "ROCBLAS_LAYER=1" } }

    accumulation_dtype:
      fp16_input: { cublas: fp32, custom_triton: fp16-or-fp32-flag, aclnn: fp32 }
      fp8_input:  { cublas: fp32, ck: fp32, aclnn: fp32 }
```

**Concrete v2.6 deliverable**: `formal_semantics` block populated on the 8 most-used operators (matmul / attention / softmax / rms-norm / silu / rope / fused-rms-residual / speculative-verify).

## Layer E — Coverage Matrix (the user's "覆盖度不同")

What it captures: a flat data-frame answering "for op X on hw Y in precision P, which library has it, with what maturity?"

This is largely **derived** from Layers C + D, but materializing it as a query-able table is what the agent actually needs.

**Schema** — auto-generated by the build at `/api/coverage-matrix.json`:

```json
{
  "rows": [
    {
      "operator": "softmax",
      "vendor": "nvidia",
      "arch": "hopper",
      "library": "cudnn",
      "precision": "fp16",
      "coverage": "full",
      "maturity": "stable",
      "deterministic": true,
      "notes": "..."
    },
    {
      "operator": "softmax",
      "vendor": "huawei",
      "arch": "ascend-910c",
      "library": "aclnn",
      "precision": "fp16",
      "coverage": "full",
      "maturity": "stable",
      "deterministic": true,
      "notes": "fused mask via aclnnMaskedSoftmax"
    },
    {
      "operator": "speculative-verify",
      "vendor": "huawei",
      "arch": "ascend-910c",
      "library": null,
      "precision": "fp16",
      "coverage": "missing",
      "maturity": "n/a",
      "notes": "No native impl; falls back to PyTorch eager (slow). Custom Ascend-C kernel needed."
    }
  ]
}
```

**This is the table an agent queries before deciding "can I deploy?"** Empty cells = either fallback path (slow but correct) or genuine gap (would need custom kernel).

**Concrete v2.6 deliverable**: `/api/coverage-matrix.json` auto-derived endpoint.

## How an agent uses these 5 layers — worked example

**Scenario**: Agent receives "deploy DeepSeek-V3 on Atlas 900 SuperPoD A2 (Ascend 910C) for chat workload at 8K req/min"

1. **Decompose model into ops** (Layer C semantic level):
   - DeepSeek-V3 = MLA-attention + GQA + RMSNorm + RoPE + SwiGLU + grouped-matmul (MoE) + speculative-verify (if MTP enabled)

2. **Query coverage matrix** (Layer E):
   - matmul (Ascend × aclnn × FP16) → ✅ full
   - rms-norm (Ascend × aclnn × FP16) → ✅ full
   - rope (Ascend × aclnn × FP16) → ✅ full
   - **MLA** (Ascend × aclnn × FP16) → ⚠️ via ATB (separate library)
   - swiglu (Ascend × aclnn × FP16) → ✅ full
   - grouped-matmul / MoE-gate (Ascend × FP16) → ⚠️ experimental (910C); ✅ stable (910D)
   - speculative-verify (Ascend × FP16) → ❌ missing

3. **Detect gap** (speculative-verify on Ascend) → **trigger codegen path**:
   - Look up Layer D `formal_semantics` for `speculative-verify` → get reference signature + edge cases
   - Look up Layer A `tensor_isa` for ascend-910c → confirm Cube + Vector units suitable
   - Look up Layer A `cross_vendor_equivalents` from existing CUDA kernel (engine_implementations[].hardware_arch contains hopper) → find primitive mapping (WGMMA → Cube)
   - Look up Layer B kernel-language → use Ascend-C with explicit GM/UB/L1 pipeline
   - Look up Layer C kernel-libraries → no existing aclnn op; emit a custom CCE-C kernel skeleton

4. **Generate** kernel from template + primitive mapping table + verified semantics

5. **Validate**: write test against Layer D edge cases; run against Layer C reference impl on Hopper for numerical equivalence

6. **Deploy**: emit final config using `/api/solve.json` recommendations + the new kernel

**Without all 5 layers, the agent stops at step 3** — it knows speculative-verify is missing but has no path forward.

## What this means in 1 sentence

> **Layer A** answers "what hardware instruction do I emit", **Layer B** answers "what language do I write the kernel in", **Layer C** answers "is there an existing library I can call instead", **Layer D** answers "if I port, will the semantics match", and **Layer E** is the materialized data-frame the agent actually queries to decide which path to take.

Without all five, an agent doing cross-hardware deployment can rank existing configurations (which is what /api/solve.json does today) but cannot autonomously generate kernel code for missing (op, hw) pairs — which is exactly the "operator generation" capability the user asked about.

---

## Implementation discipline

The schema-extension recipe (used 5× successfully — host_cpu / network_topology / storage_architecture / engine_capabilities / power_thermal) applies cleanly here:

1. Schema field additions are **all optional** → backward compatible
2. Populate **flagship entities first** (4-8 per axis) → matrix has signal even if sparse
3. Build **derived view** (`/api/coverage-matrix.json` + `/operators/coverage-matrix/`) → exposes empty cells as PR opportunities
4. **Wire into nav** → discoverability
5. **E2E tests** lock in shape

5 axes × ~3 iterations each = 15 iterations. We can ship one axis per iteration without overcommitting.

## Open questions

1. **ISA primitive normalization**: Should `WGMMA m64n128k16` and `WGMMA m64n256k16` be one entry with shape table, or two entries? Recommend: one entry with shape table — fewer YAML files, easier to maintain.
2. **Library version drift**: cuBLAS 12.0 vs 12.6 differ; aclnn evolves rapidly. Should we version `kernel-libraries/<id>.yaml`? Recommend: track *latest stable* + a `version_history` array of breaking changes.
3. **Triton special status**: Triton spans NVIDIA + AMD + Intel from one source. It's both a kernel language (Layer B) and increasingly a kernel library (Triton-flash-attn). Recommend: model as both, with cross-references.
4. **Where does Mojo / IREE / StableHLO fit?** These are *compilation targets*, not direct ISA layers. Recommend: add a sixth axis "portable IRs" later, only if usage demands.
5. **Funding the data-fill**: This is ~10× the data work of single-axis schemas. Realistic only with community PRs. Need to make individual `data/kernel-libraries/<id>.yaml` PRs as low-friction as possible.
